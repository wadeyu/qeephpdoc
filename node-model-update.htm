更新 Update

对于已有的模型对象，更新操作分为几种不同的情况：

  • 先取得要更新的对象，然后修改对象属性或调用对象方法，最后保存改动；
  • 直接更新数据库中的对象；
  • 直接更新数据库，而不涉及对象操作。

上述情况中，第二种实际上和地一种是相同的。QeePHP 会自动查询要更新的对象，然后修改属性值，最后将改动保存回数据库。

直接更新数据库，而不涉及对象操作。虽然可以获得更高的性能，但对象的业务逻辑和事件也无法处理，因此除非确实有必要，否则不应该跳过对象来直接更新数据库。

save() 方法如何确定创建或更新

不管是创建还是更新，都是调用模型的 save() 方法进行。因此 QeePHP 需要按照一定的规则来判断是进行的操作。判断规则如下：

如果对象的所有主键都有值（非 0、非空字符串、非 null），则进行更新（update）操作，否则进行（create）操作。

确保更新操作按照预期进行

许多开发者在使用 QeePHP 时，可能会犯如下的错误：

$user = new User(array('id' => 112, 'name' => 'jack', ...));
$user->save();

上面的代码从逻辑上讲很简单，但由于通常会通过 attr_protected 或 attr_accessible 设置来保护模型，因此通过构造函数是没有办法指定模型主键值的。因此上述代码的等同于：

$user = new User();
$user->name = 'jack';
...
$user->save();

最终，实际进行的将会是创建（create）操作。

 

推荐的做法：

$user = User::find('id = ?', $user_id)->getOne();
$user->age = 30;
$user->save();

上面的代码首先查询指定 id 的对象，接下来修改对象的 age 属性。调用 save() 时，将会更新指定 id 的 User 对象。

与传统的开发不同，我们这里首先进行了一次查询操作。如果对象不存在，那么查询操作将会返回一个空白对象，后续的 save() 操作就会尝试创建一个新对象。

事件

在更新一个对象时，会引发一系列的事件：

事件 说明
_before_save 保存之前
_before_update 更新记录之前
_before_validate 验证之前
_before_validate_on_update 为更新进行验证之前
_after_validate_on_update 为更新进行验证之后
_after_validate 验证之后
_after_update 更新之后
_after_save 保存之后

上述任何一个事件都可以在模型中拦截，并且通过抛出异常的方式来阻止后续事件执行。
